# ts-server-fn — 구현 계획

**위치**: `rust-sdk/packages/ts-server-fn`
**작성일**: 2026-06-13
**상태**: Draft

## 요약

dioxus-fullstack 서버 함수를 **그대로 감싸는(wrap)** proc-macro 크레이트.
`#[get]/#[post]/#[put]/#[patch]/#[delete]` 가:

1. 원본 dioxus-fullstack 서버 함수를 **변경 없이 재방출** → 서버 동작은 100% 동일.
2. 매크로 확장 시점에 `TS_SERVER_FN_PACKAGE_DIR` 가 설정돼 있으면, 해당 핸들러에 대응하는
   **TypeScript 클라이언트 함수 + 타입 import 를 `write_file` 로 디스크에 출력**.

결과: React(web) / RN(mobile) 프론트가 Rust 핸들러와 **항상 동기화되는 타입 안전 API 클라이언트**를
별도 codegen 단계 없이 빌드만으로 획득.

> 핵심 제약(사용자 요구): ts-server-fn 은 기존 dioxus server-fn 을 **wrapping** 하면서 write_file 로
> .ts 를 내보내는 형태여야 한다 → 그래야 서버 함수가 정상 동작한다.

## 근거 — 이미 존재하는 자산 (재사용)

| 파일 | 무엇 | ts-server-fn 에서의 역할 |
|---|---|---|
| `by-macros/src/server_fn.rs` | `RouteAttr` 파싱, `parse_path`, 인자 분류(client/path/query/body), extractor 제외, `Form<T>`/`Option<T>` 처리 | **파싱 로직 재사용** (공용 모듈로 추출 권장) |
| `by-macros/src/write_file.rs` | 매크로 확장 중 env-gated 파일 쓰기 | **패턴 차용** (단, `option_env!` → `std::env::var` 로 변경, 아래 주의) |
| `asset: common/fullstack/server_fn.rs` | dioxus-fullstack 와이어 계약 문서 + Rust 클라이언트 | **TS 클라이언트가 맞춰야 할 스펙** |

## 와이어 계약 (TS 클라이언트가 구현해야 할 것)

`common/fullstack/server_fn.rs` 에서 확정:

- **URL**: `{path}` — `{name}` 자리에 path 인자 치환(각각 `encodeURIComponent`),
  이어서 query `?k=v&...` (각각 인코딩, `Option` 이 `None` 이면 생략).
- **Method**: 애노테이션의 GET/POST/PUT/PATCH/DELETE.
- **Body** (POST/PUT/PATCH): **`{ "<bodyArgName>": <value> }`** — 인자 이름으로 키잉된 객체.
  (dioxus-fullstack 디코더가 `fn handler(req: T)` 를 `{"req": <T>}` 로 파싱하기 때문. 핵심.)
  body 인자 없으면 빈 body.
- **GET/DELETE**: body 없음, 입력은 path + query 로만.
- **성공 응답 (2xx)**: 반환값 `T` 의 plain JSON → `await res.json()`.
- **실패 응답 (non-2xx)**: 상태코드 + body 텍스트 → TS 는 throw.
- **인증**: same-origin 쿠키. `fetch(..., { credentials: "include" })`.
  (로컬 nginx 8080 동일 출처 → web 은 기존 쿠키 세션 그대로 사용.)

> 제약: body 인자는 **정확히 1개**만 지원 (기존 매크로와 동일). 다중 body 인자는 컴파일 에러로 거부.

## 크레이트 레이아웃

```
packages/ts-server-fn/
├── Cargo.toml              # proc-macro = true; syn, quote, proc-macro2, convert_case
└── src/
    ├── lib.rs              # #[get]/#[post]/#[put]/#[patch]/#[delete] 진입점
    ├── route.rs           # 공용 파싱 (server_fn.rs 에서 추출/공유)
    ├── write_ts.rs         # env-gated TS 파일 writer (TS_SERVER_FN_PACKAGE_DIR)
    └── tsgen/
        ├── mod.rs          # 오케스트레이션: 메타 → TS 소스 문자열 → write
        ├── type_map.rs     # Rust Type AST → TS 타입 문자열
        └── render.rs        # .ts 함수 본문 + import 렌더링
```

### 생성 산출물 구조 (OUTPUT — `$TS_SERVER_FN_PACKAGE_DIR`)

```
$TS_SERVER_FN_PACKAGE_DIR/
├── src/
│   ├── runtime/
│   │   └── client.ts       # 손으로 작성 (생성 X): fetch 래퍼·base url·credentials·
│   │                       #   error→throw·body 래핑 {argName: value}
│   ├── types/              # ts-rs 출력 (DTO·SubPartition·에러 enum)
│   │   └── *.ts
│   ├── handlers/           # ts-server-fn 생성 (핸들러당 1파일)
│   │   └── <feature>/<fnName>.ts
│   └── index.ts            # 배럴 (gen 단계에서 글롭으로 생성)
└── package.json
```

**생성(generated) vs 수기(hand-written) 분리**: 트랜스포트·인증·에러 매핑은 `runtime/client.ts` 에 한 번만
손으로 작성하고, 생성 함수는 그 위의 **얇은 타입 래퍼**만 되게 한다.

## 매크로 동작 (호출 1건당)

```rust
#[get("/api/test")]
pub async fn test_handler(user: User) -> Result<GetMeResponse> { ... }
```

1. **파싱**: route + 인자 분류 (server_fn.rs 로직 재사용).
2. **서버 함수 재방출**: 원본에 `#[::dioxus::fullstack::get("/api/test", user: User)]` 를 다시 붙여
   **변경 없이** 방출 → 서버 동작 동일. (이게 "wrap".)
3. **`std::env::var("TS_SERVER_FN_PACKAGE_DIR")` 가 Ok 면**:
   - TS 함수명 = `testHandler` (camelCase)
   - 파라미터 = `client_args` (extractor `user` 제외) → `type_map` 으로 TS 타입
   - 반환 = `Result<GetMeResponse>` 에서 `Result` 벗겨 `GetMeResponse` → `Promise<GetMeResponse>`
   - method/path/query/body 분리 → URL 빌드 + body `{ argName: value }`
   - `render` 로 .ts 문자열 생성 → `write_ts` 로 `$DIR/src/handlers/<feature>/testHandler.ts` 작성

생성 예 (개략):

```typescript
// AUTO-GENERATED by ts-server-fn. DO NOT EDIT.
import type { GetMeResponse } from "../../types/GetMeResponse";
import { apiGet } from "../../runtime/client";

export async function testHandler(): Promise<GetMeResponse> {
  return apiGet<GetMeResponse>("/api/test");
}
```

## consumer(asset) 통합

1. workspace dep `ts-server-fn` 추가.
2. `lib.rs` 의 `crate::get` 재export 심에 **세 번째 arm** 추가
   (기존: fullstack→dioxus, desktop→by_macros reqwest). 서버/fullstack 빌드에서
   `crate::get` → `ts_server_fn::get` 로 라우팅. (ts-server-fn 이 내부에서 dioxus 매크로를 재방출하므로
   서버는 그대로 동작, 추가로 .ts 만 떨어짐.)
3. 모든 wire DTO + SubPartition + 에러 enum 에 `#[derive(TS)]` 적용.
   serde 동작과 ts-rs 타입 반영이 어긋나는 지점(`SpacePartition` 같은 prefix-stripping newtype,
   serde-tagged enum)은 `#[ts(as = ..)]` / `#[ts(type = "string")]` 로 와이어 모양에 맞춤.
4. `make gen-ts` 타깃: `TS_SERVER_FN_PACKAGE_DIR=… cargo check --features server` + ts-rs export
   → 클라이언트 전체 생성.

## 생성 메커니즘 & 함정 (가장 중요한 절)

- **`option_env!` 금지, `std::env::var` 사용.** `write_file.rs` 의 `option_env!("WRITE_OUTPUT")` 는
  *by-macros 자체 컴파일 시점*에 값이 박혀서, 매크로 크레이트를 재컴파일하기 전엔 env 토글이 안 먹는다.
  proc-macro 함수 본문에서 `std::env::var(...)` 로 읽으면 **consumer 빌드의 환경**을 확장 시점에 읽으므로
  "env 설정 → consumer 재빌드 → 파일 생성" 이 의도대로 동작.
- **stale 파일**: 핸들러를 rename/삭제하면 옛 .ts 가 남는다. → `make gen-ts` 가 빌드 전에
  `$DIR/src/handlers` 를 **먼저 비우고** 매크로가 다시 채운다. (핸들러당 1파일이므로 병렬 확장 write race 없음.)
- **incremental 컴파일**: 캐시된 핸들러는 매크로가 재확장 안 됨 → .ts 재방출 안 됨. 따라서 생성은
  **매 dev 빌드가 아니라 의도적인 `make gen-ts` 패스**로 취급(클린/touch 후 실행). 문서화 필수.
- **배럴 `index.ts`**: 단일 매크로는 전체 핸들러를 못 본다 → `make gen-ts` 의 작은 스크립트가
  `handlers/**/*.ts` 글롭으로 생성.
- **타입 매핑 엣지케이스**: `Option<T>`→`T | null`, `Vec<T>`/`[T]`→`T[]`, `String/&str`→`string`,
  `i8..i64/u..`/`f32/f64`→`number`(i64 정밀도 주의, ID 는 String 이라 안전), `bool`→`boolean`,
  `HashMap<K,V>`→`Record<…>`, named→마지막 path 세그먼트 ident, `Form<T>`→`T`(기존 매크로처럼 언랩),
  `Result<T, _>`→`T`. ts-rs 의 타입명 규칙과 import 명이 일치해야 함.
- **타입 있는 에러**: Phase 2. 초기엔 `ApiError { status, body }` throw. 이후 서버가 구조화 에러 JSON
  (Translate enum → ts-rs discriminated union)을 내보내면 타입 기반 throw 로 승격.

## 테스트 계획

- **매크로 단위**: 스냅샷(또는 trybuild) — path-only / query+Option / body / extractor 제외 /
  Result 언랩 / Vec·Option 타입 각 케이스의 생성 .ts 검증.
- **라운드트립**: 샘플 핸들러로 클라이언트 생성 → 서버 기동 → 작은 vitest 가 localhost 호출 →
  디코드/throw 동작 단언.

## 리스크 / 열린 질문

- **dioxus-fullstack 와이어 포맷 고정**: 수기 `runtime/client.ts` 가 dioxus 의 요청/응답 포맷과
  계속 일치해야 함. dioxus 버전 핀 + 계약 테스트(생성 클라이언트로 실핸들러 호출)로 가드.
- **ts-rs ↔ serde 괴리**: SubPartition / tagged enum 에 `#[ts]` 오버라이드 작업량 선반영.
- **body 1개 제약**: 다중 body 인자는 컴파일 에러로 거부 (기존 매크로와 동일 정책).
- **incremental 누락**: 생성은 클린 패스 전제 — 부분 빌드에서 일부 .ts 누락 가능. `make gen-ts` 가 유일한 진실.
